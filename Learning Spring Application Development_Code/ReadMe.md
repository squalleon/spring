## 요약
이 책은 스프링 입문서로, 스프링의 핵심 개념이라고 할 수 있는 제어역전(IoC)과 관점지향 프로그래밍(AOP), MVC 모델은 물론이고 ORM 기반의 데이터베이스 연동, 테스트, 메일 서버 연동 등 애플리케이션 개발에 필수적인 내용을 담고 있다. 특히 스프링 4.x의 새로운 기능을 소개해 스프링에 이미 익숙한 개발자에게도 유용한 정보를 제공한다.

■ 스프링 컨테이너 설정 및 XML과 애노테이션을 이용한 스프링 빈 관리

■ 스프링 시큐리티를 이용해 불법적인 침입자로부터 애플리케이션을 보호하는 방법

## 이 책의 대상 독자
이 책은 스프링 프레임워크를 배우고자 하는 사람들을 대상으로 하며, 자바 프로그래밍과 웹 애플리케이션에 대한 사전지식이 필요하다. 개발자든, 아키텍트든, 관리자든 팀에서의 역할에 상관없이, 이 책은 적용 가능한 스프링 기술을 학습할 수 있도록 가장 효과적이고 적절한 방법을 제공할 것이다. XML에 익숙하면 좋겠지만 XML 초보자도 큰 어려움 없이 이 책의 내용을 이해할 수 있다. 또한 엔터프라이즈 개발 지식이 있으면 좋겠지만 필수는 아니다. 이 책은 코어 층, 데이터 접근 계층, 웹 계층에 기반한다. 개발 예제 코드를 통해 단계적으로 접근해 초보자도 쉽게 애플리케이션 개발을 이해할 수 있다.

## 이 책의 구성
1장, ‘스프링 프레임워크 소개’에서는 스프링 프레임워크 구조에 대한 이해와 스프링 애플리케이션 개발 환경의 주요 컴포넌트 설정을 살펴본다. 이 책의 나머지 부분에 대한 로드맵 역할을 한다.

2장, ‘제어역전’에서는 스프링 컨테이너 구성과 XML을 통한 스프링 빈 관리에 대해 알아본다. 제어역전(IoC, Inversion of Control)과 의존성 주입(DI, Dependency Injection)의 개념을 살펴본다.

3장, ‘스프링에서의 DAO와 JDBC’에서는 DAO 디자인 패턴과 스프링을 이용해 데이터에 대한 접근 권한을 부여한다. 스프링 프레임워크에 JDBC 지원과 ORM 지원을 구현한다. 스프링이 데이터 자원을 관리하는 방법과 애플리케이션에서 사용 가능한 데이터 자원은 무엇인지 살펴본다.

4장, ‘스프링과 하이버네이트 연동’에서는 스프링에서 폭넓게 지원되는 ORM 라이브러리 중 하나인 하이버네이트(Hibernate)를 살펴본다. 매핑 설정과 영속 클래스의 매핑을 살펴보고 스프링 애플리케이션에서 어떻게 하이버네이트가 작동하도록 구성하는지 알아본다.

5장, ‘스프링 웹 MVC 프레임워크’에서는 스프링 웹 컨트롤러와 스프링 폼 태그 라이브러리를 활용해 스프링 MVC 애플리케이션을 생성한다. 스프링 MVC를 소개하고 스프링 MVC가 제공하는 강력한 기능을 통해 어떻게 고성능 웹 애플리케이션을 개발하는지 설명한다.

6장, ‘스프링 시큐리티를 이용한 보안’에서는 스프링 시큐리티를 통해 애플리케이션을 악의적인 침입자로부터 보호할 수 있다. Acegi Security System을 소개하고 서블릿 필터를 통한 웹 애플리케이션 보안을 살펴본다.

7장, ‘스프링을 이용한 테스트’에서는 Junit과 TestNG를 사용해 실제 테스트 전략을 구현한다. Junit 프레임워크에 집중해 단위 테스트 실행 방법을 설명한다.

8장, ‘JavaMail과 JMS의 스프링 연동’에서는 이메일 송수신을 위한 스프링 메일 애플리케이션 프로그래밍 인터페이스를 구현한다. 비동기 처리를 위한 JMS(Java Messaging Service)를 소개한다.

9장, ‘애노테이션을 이용한 스프링 제어역전’에서는 애노테이션을 통해 스프링 빈과 의존성 주입을 구성한다. 애노테이션 기반의 의존성 주입과 애노테이션 생명주기를 살펴본다. SpEL(Spring Expression Language)을 이용한 빈 참조와 메소드 호출, SpEL의 연산자를 살펴본다. 또한 스프링이 제공하는 텍스트 메시지와 국제화를 살펴보고, 애플리케이션 구현을 통해 학습한다.

10장, ‘스프링의 관점지향 프로그래밍’에서는 관점지향 프로그래밍(AOP)을 소개한다. 스프링의 강력한 포인트컷 메커니즘을 이용해 애플리케이션에 애스펙트(aspect)를 어디에 어떻게 적용하는지 살펴보고 스프링 AOP의 프록시를 살펴본다.

부록 A, ‘연습문제 해답’에서는 이 책의 모든 연습문제 해답을 제공한다.

부록 B, ‘아파치 더비를 이용한 애플리케이션 데이터베이스 구성’에서는 아파치 더비(Derby) 애플리케이션 데이터베이스 설정 방법을 알려준다.

부록 C, ‘스프링 폼 태그 라이브러리’에서는 스프링 웹 MVC 프레임워크가 제공하는 스프링 폼 태그 라이브러리를 살펴본다. 스프링 폼 태그 라이브러리는 폼 태그 라이브러리 내부의 태그 집합으로, 뷰(웹 페이지)를 구축하는 데 사용된다.

- 지은이의 말
스프링 프레임워크는 자바 애플리케이션 개발에 광범위한 인프라 지원을 제공하는 최첨단 프레임워크다. 스프링 프레임워크는 개발자가 애플리케이션에 집중할 수 있도록 인프라를 처리한다. 이는 POJO 기반 프로그래밍 모델을 가능하게 함으로써 좋은 프로그래밍을 실천하도록 장려하고 애플리케이션을 계층화해 구성하는 훌륭한 방법을 제공한다. 대규모의 자바 개발 프로젝트를 설계, 디자인, 개발하는 경우 성능이 좋고 테스트가 용이하며 재사용 가능한 코드를 생성하는 스프링 프레임워크를 활용하는 것이 적절하다.
어떤 사람들은 읽으면서 배우는 것을 선호하는 반면, 어떤 사람들은 코딩하면서 배우는 것을 선호한다. 나는 코딩을 통해 배우는 것이 더 좋은 결과를 낳는다고 믿고, 이 책에서도 그 방식을 선택했다. 스프링 프레임워크의 각 기능에 대한 이해를 돕기 위해 풍부한 예제 코드와 적절한 설명을 담았다. 여러분은 첫 장에서부터 스프링 프레임워크를 사용해 애플리케이션 개발을 시작할 수 있을 것이다.
스프링 프레임워크는 수많은 기능을 담고 있는 바다다. 이 책은 애플리케이션에서 일반적으로 사용되는 다양한 기능을 다루고 각 기능을 코드 기반의 예제를 통해 살펴본다. 코드와 도표를 통해 내용을 보충하며 추가적으로 라이브러리 스냅샷을 통해 각 예제와 결과를 제공한다.

## 부록 A 연습문제 해답

## 1장. 스프링 프레임워크 소개

### Q1. 스프링이랑 무엇인가?
스프링은 로드 존슨 Rod Johnson이 만든 오픈소스 프레임워크다. 로드 존슨은 자신의 저서 "J2EE 설계와 개발 expert one-on-one" (정보문화사, 2004) 에서 엔터프라이즈 애플리케이션 개발의 복잡성을 해결할 수 있는 간단한 대안을 제시했다.
슾츠링은 제어 역전 Inversion of Control을 지원하는 경량의 관점지향 aspect-oriented 컨테이너 프레임워크다.
스프링 프레임워크 기반의 자바 EE 애플리케이션 개발은 단순성, 느슨한 결합, 테스트 용이성 측면에서 유용하다.
스프링은 추가적인 복잡한 과정 없이 필요한 부분을 갖다 쓸 수 있게 모듈화돼 있으며, 전체 계층을 구현하는 일은 물론 특정 계층을 구현하는 일에도 사용할 수 있다.

### Q2. 스프링의 특징을나열하라.
스프링 프레임워크는 다음과 같은 특징이 있다.
- 경량화 Lightweight: 스프링은 사이즈와 투명성(자유롭게 가져다 쓸 수 있는) 측면에서 경량 프레임워크다.
- 비침투성 Non-intrusive: 비침투성이란 도메인 로직 코드가 프레임워크에 의존적이지 않음을 의미하나다. 스프링 프레임워크는 비침투성을 갖도록 설계됐다.
- 컨테이너 Container: 스프링 프레임워크는 경량 컨테이너로서 애플리케이션 객체의 생명주기와 설정을 관리한다.
- 제어역전 Inversion of Control: 제어역전이란 의존성 주입을 설명하는 아키텍처 패턴으로, 컴포넌트 스스로 의존성을 생성하지 않고 외부 엔티티에 의해 의존성이 주입된다.
- 관점지향 프로그래밍 AOP, Aspect-oriented Programming: AOP란 메인 프로그램의 비지니스 로직 기능과 지원 기능을 분리하는 일종의 프로그래밍 패러다임이다.
- JDBC 예외 처리: 스프링 프레임워크의 JDBC 추상 계층은 계층구조의 예외를 제공한다.
- 스프링 MVC 프레임워크: 강력하고 유지보수가 용이한 웹 애플리케이션 개발을 돕는다. 스프링 웹 MVC 프레임워크는 웹 애플리케이션 개발에서 가장 범용적으로 쓰이는 유티리티 클래스르 제공한다.
- 스프링 보안: 스프링 보안은 스프링 기반 애플리케이션 대다수에서 중요한 부분인 선언적 보안 메커니즘을 제공한다.

### Q3. 스프링 프레임워크의 각 모듈을 설명하라.
스프링 프레임워크는 다음과 같은 모듈을 포함한다.
- 스프링 코어 모듈
- 스프링 AOP 모듈
- 스프링 DAO(JDBC) 모듈
- 스프링 ORM 모듈
- 스프링 웹 모듈
- 스프링 테스트 모듈

## 2장. 제어역전
### Q1. 제어역전과 의존성 주입이란 무엇인가?
제어역전은 좀 더 일반적인 개념이며, 반면에 의존성 주입은 구체적인 디자인 패턴이다.
소프트웨어 엔지니어링 관점에서 제어역전이란 정적 분석을 사용해 런타임 시 어셈블러 객체에 의해 객체 간 결합이 일어나는 프로그래밍 기법의 하나다.
의존성 주입은 스스로 객체에 의해 생성되는 것이 아니라 프레임워크에 의해 객체 의존성이 주입되는 디자인 패턴으로 객체 사이의 결합도coupling 을 줄여준다.
제어역전은 코드를 좀 더 관리가 쉽고 더 테스트 용이하며 더 이식하기 쉽게 해준다. 또한 컴포넌트 설정, 의존성, 생명주기, 이벤트를 컴포넌트와 분리시킨다.

### Q2. 스프링의 의존성 주입의 종류는 무엇인가?
- 생성자constructor 주입: 다수의 인자를 포함한 생성자를 호출함으로써 생성자 기반의 의존성 주입을 수행한다. 인스턴스를 인스턴스화할 때 인자가 주입된다.
- 세터setter 주입: 세터 기반의 의존성 주입은 스프링에서 선호되는 의존성 주입 메소드로, 인자가 없는 정적 팩토리 메소드 또는 인자가 없는 생성자가 빈에 인스턴스화하기 위해 호출된 후, 빈에 세터 메소드를 호출함으로써 수행된다.

### Q3. 스프링의 오토와이어링을 설명하라. 오토와이어링의 종료는 무엇인가?
다음과 같이 스프링 컨테이너는 다섯 가지의 오토와이어링 모드를 사용할 수 있다.
- no: 스프링 빈 오토와이어링의 기본값은 사용 안 함turned off이다. 오토와이어링을 사용하지 않는다면 연결을 위해 명시적으로 빈 참조 ref를 사용해야 한다.
- byName: 속성명에 의해 오토와이어링된다. 즉 빈 속성명이 같다면 자동 연결한다. 종석성 주입을 위한 byName 형태의 오토와이어링에는 세터 메소드가 사용된다.
- byType: 데이터 다임이 오토와이어링에 사용된다. 만약 데이터 타입 빈 속성이 다른 빈의 데이터 타입과 호환되면, 자동 연결된다. 하나의 빈만이 설정 파일에서 byName 모드로 구성하고 하나 이상일 때는 치명적인 예외가 발생한다.
- constructor: byType 모드와 비슷하지만 생성자 모드는 의존성 주입 시 사용된다.
- autodetect: autodetect 모드는 더 이상 지원하지 않는다. 스프링은 생성자에 의한 자동 연결을 먼저 시도하지만 안 될 경우 다음으로 스프링은 byType 모드의 자동 연결을 시도한다.

### Q4. 스프링 빈 스코프 각각의 차이점을 설명하라.
다음은 스프링 빈 스코프 목록이다.
- 싱글톤Singleton: 스프링 싱글톤은 특정 스프링 컨테이너를 의미하고 다른 참조에 사용하는 컨테이너에서 생성된 빈 인스턴스는 하나만 존재한다.
- 프로토타입Prototype: 요청 또는 참조마다 새로운 빈을 생성한다. 스프링은 초기화를 수행하고 따라서 컨텍스트가 생성되는 동안 기본값 초기화 대신 getBean() 호출을 기다린다.
- 요청Request: 서블릿 요청마다 새로운 빈이 생성된다. 스프링은 서블린 API와 밀접히 묶여 있기 때문에 새로운 요청이 발생할 때를 인지하고 요청에 따라 새로운 빈을 생성한다.
- 세션Session: 세션마다 새로운 빈이 생성된다. 단일 세션에 접근하는 하나의 사용자만 있는 한 getBean()을 호출할 때마다 같은 빈 인스턴스를 반환한다.
- 글로벌 세션Global session: 포틀릿 컨텍스틑 환경에서 적용된다. 글로벌 세션은 개별 프틀릿 세션에 존재하고 빈은 글로벌 세션과 묶인다. 글로벌 HTTP 세션마다 새롱누 빈이 생성된다.

## 3장. 스프링에서의 DAO와 JDBC
### Q1. 스프링 JDBC 패키지에 대해 설명하라.
JDBC의 다양한 부분을 처리하기 위해 스프링 JDBC는 다음 표와 같은 패키지로 구분된다.

스프링 JDBC 패키지	            설명
org.springframework.jdbc.core	스프링 프레임워크에서 org.springframework.jdbc.core 패키지는 코어 JDBC 클래스와 JdbcTemplate을 포함한 JDBC 클래스 파운데이션을 포함한다.
org.springframework.jdbc.datasource org.springframework.jdbc.datasource 패키지는 DataSource 구현과 JEE 컨테이너 외부에서 JDBC 코드를 실행시켜주는 도움 클래스(helper classes)를 포함한다.
org.springframework.jdbc.object     스프링 프레임워크에서 org.springframework.jdbc.object 패키지는 데이터베이스에서 일반 자바 객체(plain Java objects)로 반환된 데이터의 변환을 도와주는 클래스르 포함한다.

org.springframework.jdbc.config     org.springframework.jdbc.config 패키지는 스프링 프레임웤의 ApplicationContext와 함께 JDBC 설정을 지원하는 클래스르 포함한다.

### Q2. JdbcTemplate이란 무엇인가?
JdbcTempate은 여러 종류의 SQL 구문을 실행하는 데 사용된다. DML Data Manipulation Language 문을 사용해 데이터베이스로 데이터를 추가, 조회, 수정, 삭제한다.
DML 문의 예로는 SELECT 또는 INSERT, UPDATE 문이 있다. DDL Data Definition Language 문을 사용해 데이터베이스의  데이터 객체의 구조를 생성 또는 변경한다.
DDL 문의 예로는 CREATE, ALTER, DROP 문이 있다.

### Q3. 스프링에서의 JDBC 일괄작업을 설명하라.
여러 작업을 수행하기 위한 하나의 실행 유닛을 일괄작업이라 한다. 수정 일괄 작업은 다수의 SQL 질의 DataSource로 제출해 한 번에 처리한다.
다수의 질의를 개별로 보내지 않고 한 번에 보냄으로써 성능이 향상된다. JdbcTemplate은 JDBC 구문과 PreparedStatement를 통해 일괄작업 구문 수행을 지원한다.
JdbcTemplate은 이러한 기능을 지원하는 두 종류의 오버로드 메소드인 batchUpdate()를 갖는다.
- 첫 번째 메소드는 JDBC 구문을 사용해 SQL 문을 일괄적업으로 실행한다. 이러한 메소드의 특징은 다음과 같이 데이터 갱신을 유발하는 SQL을 여러 번 실행시킨다는 것이다.
  public int[] batchUpdate(String[] sql) throws DataAccessException

- 두 번째 메소드는 PreparedStatement를 사용해 서로 다른 매개변수와 함께 SQL 문을 여러 번 수행한다. 다음 코드를 살펴보자.
  public int[] batchUpdate(String sql, BatchPreparedStatementSetter bPSS) throws DataAccessException

## 4장. 스프링과 하이버네이트 연동
### Q1. ORM이란 무엇인가?
ORM은 RDBMS를 비롯한 관계형 데이터베이스에 객체를 영속적으로 저장하는 과정을 말한다.
ORM은 객체 스키마와 관계형 스키마 사이의 간극을 메유는 역할을 하며, 객체를 관계형 형식으로 변환하지 않고도 객체지향 애플리케이션이 객체를 직접 영속적으로 저장할 수 있게 한다.
ORM은 객체 표현을 JDBC 구문의 매개변수로 매핑하고 다시 JDBC 질의 결과를 객체 표현으로 매핑하는 과정을 통칭한다.
이때, 데이터베이스의 컬럼은 도메인 객체 인스턴스의 필드나 자바 빈의 프로퍼티와 매핑된다.

### Q2. 하아버네이트 아키텍처의 기본 구성요소를 설명하리.
하이버네이트 하키텍처의 기본 구성요소를 설명하면 다음과 같다.
- Configuration: org.hibernate.cfg.Configuration 클래스는 SessionFactory를 생성하는데 필요한 하이버네이트 API의 기본 구성요소다. 즉, Configuration은 SessionFactory를 생성하는 팩토리 클래스라고 할 수 있다.
- SessionFactory: SessionFactory는 애플리케이션 초기 구동 시 생성돼, 애플리케이션에서 추후에 사용하기 위해 유지된다. org.hibernate.SessionFactory 인터페이는 하이버네이트 세션 객체를 얻는 데 필요한 추상화를 제공하는 팰토리 역할을한다. SessionFactory의 초기화 과정은 많은 자원과 시간을 소모하는 여러 가지 연산을 포함하므로, JVM 인스턴스 하나당 SessionFactory 하나를 사용하기를 권장한다.
- Session: org.hibernate.Session은 하이버네이트 시스템과 애플리케이션 사이의 인터페이스로서, 데이터베이스 커넥션을 얻는 데 사용한다.
  세션은 경량 객체로, 데이터베이스와의 상호작용이 필요할 때마다 초기화된다. 그리고 Session의 사용이 모두 끝난 후에는, 캐싱된 엔티티 객체나 JDBC 커넥션을 비롯한 자원을 해제할 수 있도록 세션을 닫아야 한다.
- Transaction: 트랜잭션 인터페이스는 대부분의 RDBMS에서 지원되며, 데이터베이스의 작업 단위 하나를 표현한다. 하이버네이트의 Transaction은 기반 트랜잭션 관리자를 바탕으로 처리된다.
- Query: org.hibernate.Query 인터페이스는 하이버네이트 질의를 실행하고 결과를 조회할 때 필요한 추상화를 제공한다.
  Query 객체는 하이버네이트 질의 언어로 만들어진 하이버네이트 질의를 표현한다.
- Criteria: org.hibernate.Criteria 인터페이스는 기준 API를 위한 것으로 HQL이나 SQL의 대안인 객체지향 기준 질의를 생성하고 실행할 때 사용한다.
- Persistent: 이 클래스는 애플리케이션의 엔티티 클래스로, 영속성 객체란 그 상태를 영속적으로 관리하는 클래스를 말한다.
  영속석 객체는 하나의 org.hibernate.Session에만 연관되며, org.hibernate.Session이 닫힐 때 연관된 영속성 객체도 분리돼 애플리케이션의 어느 계층에서나 사용할 수 있게 된다.

### Q3. HQL이란 무엇인가?
하이버네티트 질의 언어 HQL는 테이블과 컬럼이 아니라 영속성 객체의 그 프로퍼티를 대상으로 작동하는 객체지향 질의 언어다.
하이버네이트는 데이터베이스와 상호작용하는 과정에서 HQL 질의를 전통적이 SQL 질의로 변환한다. HQL에서 SELECT와 FROM, WHERE, GROUP BY를 비롯한 키워드에는 대소문자 구분이 없다.

## 5장. 스프링 웹 MVC 프레임워크
### Q1. 스프링 웹 MVC 프레임워크란 무엇인가?
스프링 웹 MVC Model View Controller 프레임워크는 유연하고 강력하며 잘 설계됐으며, 웹 애플리케이션 개발에 사용된다.
프레임워크는 웹 애플리케이션 개발을 모델, 뷰, 컨트롤러로 구성할 수 있게끔 설계됐다.
스프링 웹 MVC 프레임워크는 자바, 서블릿, JSP와 같은 가자 기술을 사용해 구현됐다.
따라서 스프링 JAR 파일을 웹 애플리케이션/프로젝트의 lib 폴더에 위치시킴으로써 모든 자바 엔터프라이즈 웹 서버상의 스프링 MVC 프로젝트를 호스팅할 수 있다.
스프링 프레임워크의 스프링 MVC 모듈은 국제화와 테마, 검증 등 표현 계층 구현을 돕는 기능인 MVC 디자인과 같은 포괄적인 지원을 제공한다.
스프링 MVC 프레임워크는 DispatcherServlet을 중심으로 설계됐다.
DispatcherServlet은 핸들러에게 HTTP 요청을 전달하는 매우 단순한 컨트롤러 인터페이스다.

### Q2. 스프링 MVC 프레임워크의 DispatcherServlet이란 무엇인가?
스프링 MVC 프레임워크의 DispatcherServlet 클래스는 프론트 컨트롤러의 구현이며 스프링 MVC 애플리케이션의 자바 서블릿 컴포넌트다.
DispatcherServlet은 스프링 MVC 애플리케이션으로 들어오는 모든 HTTP 클라이언트 요청을 받는 프론트 컨트롤러 클래스이자, 다양한 단계의 요청 처리를 하는 프레임워크 컴포넌트를 초기화하는 역할을 한다.
DispatcherServlet 클래스는 스프링 컨텍스트와 스프링 ORM Object Relational Mapping, 스프링 DAO Data Access Object 등과 같은 다양한 스프링 기능 사용을 위한 제어역전 컨테이너와 완벽히 구성된다.
DispatcherServlet 은 HTTP 요청을 처리하는 서블릿이며 HttpServlet 베이스 클래스르 상속받는다.
스프링 MVC 애플리케이션은 DispatcherServlet 클래스를 여러 개 가질 수 있고 각 DispatcherServlet 클래스는 자신의 WebApplicationContext를 갖는다.

### Q3. 스프링 MVC에서 컨트롤러란 무엇인가?
DispatcherServlet은 들어오는 HTTP 클라이언트 요청을 특정 기능을 수행하도록 컨트롤러에게 위임한다.
컨트롤러는 사용자 입력을 해석하고 입력을 사용자 뷰로 표현되는 특정 모델로 변환한다.
특정 인터페이스 또는 클래스를 상속받을 필요없이 @Controller 애노테이션을 사용해 컨트롤러 클래스를 정의할 수 있다.

### Q4. 스프링 MVC에서 ViewResolver란 무엇인가?
컨틀롤러 클래스 핸들러 메소드는 논리 뷰명을 의미하는 다른 값을 반환한다. 뷰는 JSP, FreeMarker, PDF, 엑셀, XSL 페이지를 의미한다.
DispatcherServlet 으로부터 뷰 템플릿으로 제어가 위임된다. 메소드에 의해 반환된 뷰명은 웹 애플리케이션 컨텍스트상에 선언된 ViewResolver 빈에 의해 실제 논리 소스로 고정된다.
스프링은 XML 파일상에 구성되는 다수의 ViewResolver 클래스를 제공한다.
ViewResolver 인터페이스는 org.springframework.web.servlet.ViewResolver 인터페이스의 구현과 뷰명을 매핑한다.

## 6장. 스프링 시큐리티를 이용한 보안
### Q1. 스프링 시큐리티란 무엇인가?
스프링 시큐리티 프레임워크는 스프링 기반 애플리케이션을 보호하기 위한 사실상의 표준으로, 인증과 권한 관리를 바탕으로 엔터프라이즈 자바 소프트웨어 애클리케이션에 대한 보안 서비스르 제공한다.
스프링 시큐리티는 웹 요청 레벨과 메소드 호출 레벨 모두에서 인증과 권한 처리를 지원한다.
스프링 시큐리티는 높은 수준의 맞춤화가 가능하며, 강력한 인증과 접근 제어를 제공한다.

### Q2. 인증과 권한은 무엇인가?
인증은 사용자의 신원이 정확한지 확인하는 과정으로, 식별과 검증으로 이뤄진다.
식별을수행하는 방법에는 여러 가지가 있는데, 예를 들어 사용자명과 비밀번호를 이용하거나 이를 데이터베이스에 저장하거나, LDAP, CAS(통합인증 프로토콜) 등을 이용할 수 있다.
권한 관리는 인증된 사용자에 대한 접근 제어를 제공하며, 인증된 특정 사용자가 허가된 사용자만 사용할 수 있는 자원에 대한 접근 제한이 있는지를 확인하는 과정이다.

### Q3. 스프링 시큐리티가 지원하는 여러 가지 웹 애플리케이션 로그인 방식들의 차이점은 무엇인가?
스프링 시큐리티가 지원하는 여러 가지 웹 애플리케이션 로그인 방식은 다음과 같다.
- HTTP 기본 인증: 스프링 시큐리티가 지원하는 HTTP 기본 인증은 HTTP 헤더에 포함된 기본 계정 정보를 처리한다. HTTP 기본 인증은 요청마다 계정 정보를 전달하는 상태 비보존 클라이언트에서 주로 사용한다.
- 폼 기반 로그인 서비스: 스프링 시큐리티에서 지원하는 폼 기반 로그인 서비스는 사용자가 웹 애플리케이션에 로그인하는 데 필요한 기본 로그인 폼 페이지를 제공한다.
- 익명 로그인: 스프링 시큐리티에서 지원하는 익명 로그인 서비스는 일반 사용자를 비롯한 익명 사용자의 인증을 제공한다.
- 로그인 정보 기억하기 지원: 스프링 시큐리티가 지원하는 로그인 정보 기억은 여러 브라우저 세션에 걸쳐 사용자의 신원을 기억한다.

## 7장. 스프링을 이용한 테스트
### Q1. JUit 4와 TestNG의 차이점은 무엇인가?
JUnit과 TestNG는 모두 단위 테스트 프레임워크로 기능 면에서는 매우 유사하다.
두 프레임워크 모두 애노테이션 지원과 예외 테스트, 무시 ignore 테스트, 통합 테스트 등의 기능을 제공한다.
반면 그룹 테스트와 의존성 테스트는 TestNG에서만 지원한다.
TestNG는 JUnit과 달리 매개변수화된 테스트에서 사용하기 위한 테스트 데이터를 동적으로 생성할 수 있다.
다음 표는 TestNG와 JUnit4에서 지원하는 애노테이션의 일부를 보여준다.

기능                                   TestNG              JUnit4
테스트 애노테이션                        @Test           @Test
현재 클래스의 첫 번째 테스트 메소드 이전    @BeforeClass    @BeforeClass
현재 클래스의 모든 테스트 메소드 이후       @AfterClass     @AfterClass
각 테스트 메소드 이전                    @BeforeMethod   @Before
각 테스트 메소드 이후                    @AfterMethod    @After
현재 스윗 안의 모든 테스트 이전           @BeforeSuite
현재 스윗 안의 모든 테스트 이후           @AfterSuite
테스트 수행 이전                       @BeforeTest
테스트 수행 이후                       @AfterTest

### Q2. 단위 테스트와 통합 테스트의 차이점은 무엇인가?
단위 테스트라는 이름에서 알 수 있듯이 코드를 구성한느 모든 개별 메소들 테스트하는 활동을 말한다.
즉, 긴을 동작하게 하는 기본적인 요소들을 테스트한다.
한편으로는 특정 기능을 구현한 코드를 테스트할 목적으로 소프트웨어 개발자가 작성한 코드를 단위 테스트라고 할 수 있다.
테스트는 버그를 찾기 위해서가 아니라 ,코드의 품질으 향상시키고 버그를 예방하는 목적으로 사용한다.
단위 테스트는 테스트 프레임워크를 이용해 자동화된다.
통합 테스트는 소프트웨어 테스트의 한 단계로, 개별적인 소프트웨어 모듈을 조합해 하나의 그룹으로 테스트함으로써 해당 단위들이 제대로 통합돼 올바르게 상호작용하는지를 확인하는 작업이다.
통합 테스트의 목적은 코드의 기능성과 성능, 안정성를 검증하는 데 있으며, 어려 단위를 함께 테스트해야 하는 경우에 통합 테스트를 활용한다.

### Q3. 스프링 MVC 테스트 프레임워크를 설명하라.
스프링 MVC 테스트 프레임워크는 뛰어난 JUnit 지원을 바틍으로 스프링 MVC 컨트롤러의 좀 더 의미 있는 단위 테스트와 통합 테스트를 가능하게 한다.
예전에는 테스트할 수 없었던 컨트롤러 메소드의 모든 측면을 테스트하는 데 도움으 주며, 웹 컨테어이늘 구동하지 않고도 그러한 측면을 테스트할 수 있다.
스프링 TestContext 프레임워크에 JUnit이나 TestNG를 함께 사용하면 애노테이션 기반 단위 테스트와 통합 테스트 지원을 바탕으로 스프링 MVC 프레임워크의 테스트를 간단히 수행할 수 있다.
스프링 TestContext 프레임워크는 @RunWith와 @WebAppConfiguration, @ContextConfiguration 등의 애노테이션을 이용해 테스트를 수행하며,
이러한 애노테이션은 스프링 설정을 불러오고 단위 테스트와 통합 테스트에서 사용할 WebApplicationContext를 MockMvc에 주입한다.

### Q4. @ContextConfiguration와 @WebAppConfiguration을 설명하라.
@ContextConfiguration 애노테이션은 주어진 파일 경로에서 실제 설정 파일을 읽어와서 테스트 클래스에 ApplicationContext를 설정한다.
코드에서 파일을 지정했는데, 이렇게 지정한 상대 경로를 루트 패키지로 사용한다. 파일의 경로를 지정하면 정확한 위치를 정해줄 수도 있다.
그리고 다음과 같이 쉼표를 구붅다로 이용해 하나 이상의 설정 파일을 전달할 수 있다.
@ContextConfiguration 애노테이션은 ApplicationContext를 캐싱해, 테스트를 수행하거나 테스트 스윗이 동작하는 동안 정적 메모리에 저장한다.
ApplicationContext를 정적 메모에 저장하면 동일한 JVM에서 수행되는 모든 테스트는 해당 ApplicationContext를 사용한다.
두 번째 JVM이 존재한다면 앞서 만든 정적인 컨텍스트에 접근할 수 없으므로 새로운 ApplicationContext가 만들어진다.
클래스 레벨 애노테이션인 @WebAppConfiguration을 이용하면 스프링 프레임워크에서 웹용 애플리케이션 컨텍스트를 만들 수 있다.
즉, 통합 테스트를 위해 만들어진 ApplicationContext 클래스 객체가 WebApplicationContext의 인스턴스임의 알리는 역할을 한다.
@WebAppConfiguration 애노테이션은 @ContextConfiguration 애노테이션과 함께 사용해야 한다는 점을 알아두자.

### Q5. MockMvc와 @RunWith(SpringJUnit4ClassRunner.class)를 설명하라.
MockMvc는 스프링 MVC 테스트 프레임워크의 핵심적인 부분으로, 스프링 MVC를 이용해 만들어진 애플리케이션의 테스트를 작성하는 데 사용할 수 있다.
스프링 MVC 테스트의 진입점 역할을 하는 MockMvc는 스프링 MVC의 기반 구조 전체를 대상으로 모의 객체를 생성할 수 있으며,
스프링 MVC 테스트 프레임워크를 사용하려면 우선 MockMvcBuilder 의 구현을 이용해 MockMvc 인스턴스를 만들어야 한다.
@RunWith 애노테이션은 JUnit 애노테이션으로 @RunWith 애노테이션이 달린 클래스 안에서 테스트를 실행하거나, 매개변수로 전달된 클래스를 호출해
@RunWith 애노테이션이 부여된 클래스를 확장한다.
즉, 이 애노테이션이 부여된 클래스 안의 테스트는 JUnit 프레임워크에 내장된 APIFNF를 이용해 수행되는 대신,
지정한 러너 runner 클래스를 이용해 테스트를 수행한다.
스프링 ApplicationContext 환경에서 스프링 JUnit 클래스 러너를 이용해 테스트 케이스를 수행하려면
스프링의 SpringJUnit4ClassRunner 클래스르 매개변수로 전달한다.

## 8장. JavaMail과 JMS의 스프링 연동
### Q1. JavaMail API는 무엇인가?
JavaMail API는 자바 애플리케이션을 위해서 프로토콜 독립적이고 플랫폼 독립적인 프레임워크를 제공한다.
JavaMail API는 이메일 시스템을 구성하는데 필요한 클리스와 인터페이스의 모음이다.
JavaMail API를 이용해 간단한 이메일을 보내는 과정은 다음과 같다.
1) 사용자명과 패스워드를 지정해 이메일 서버에 연결한다.
   예를 들어, abc@xyz.com에서 이메일을 보내려 한다면 이메일 서버인 xyz.com에 연결해야 한다.
2) 참조 Cc와 숨은 참조 Bcc를 비롯한 수신자 참조를 지정해 메시지를 생성한다.
3) 첨부가 존재하면 메시지에 추가한다.
4) 메시지를 이메일 서버에 전송한다.

### Q2. 메시지와 메시징은 무엇인가?
메시지는 두 종단점 간에 주고 받는 바이트로 이뤄진 데이터나 정보를 말한다.
여러 규약에 따라 메시지를 다른 방식으로 정의할 수 있으나, 통신의 한 요소라는 점에는 변함이 없다.
메시지는 한 애플리케이션에서 다른 애플리케이션으로 정보를 전달할 때 사용하며,
이때 두 애플리케이션은 동일한 플랫폼에서 구동될 수도 있고 서로 다른 플랫폼에서 구동될 수도 있다.
메시징은 (분산 환경에) 서로 다른 애플리케이션이나 시스템 구성요소 사이의 통신을 말한다.
이때 각 구성요소는 TCP 소켓의 두 피어처럼 강한 결합이나 원격 메소드 호출 RMI, Remote Method Invocation과 CORBA를 비롯한 느슨한 결합을 모두 포함한다.
자바 메시징의 장점으로는 서로 다른 플랫폼 간의 통합과 메시지 전달의 확장성과 안정성 향상, 시스템 병목의 감소 등을 들 수 있다.
이처럼 메시징을 이용하면 메시지를 소비하고 생산한느 시스템과 클라이언트를 원하는 만큼 확장할 수 있다.

### Q3. JMS는 무엇인가?
자바 메시지 서비스 JMS, Java Message Service는 자바 메시지 지향 미들웨어 MOM, Message Oriented Middleware의 API로,
둘 이상의 클라이언트 사이에 메시지를 보낼 때 사용한다.
즉, JMS는 메시지 발신자와 수신자 사이에서 중재자 역할을 하는 우편배달부처럼 중계자 임무를 수행한다.
JMS는 자바 프로그램에서 분산된 기업환경의 메시지를 생성하고 전송하고 읽을 수 있는 공통된 방식을 기술하는 규약이며,
발신자와 수신자를 고려할 필요가 없는 느슨한 결합으 지향한다. 또한 비동기 메시징을 지원하는데,
이는 발신자와 수신자가 동시에 존재할 수 있고 그렇지 않을 수도 있음을 뜻한다.
즉 메시지를 보내거나 받는 두 시스템이 동시에 구동돼야 할 필요가 없다.

## 9장. 애노테이션을 이용한 스프링 제어역전
### Q1. 스테레오 타입 애노테이션은 무엇인가?
@Component(구성요소)는 부모 스테레오 타입 애노테이션으로 일반적으로 모든 빈을 정의한다.
이 구성요소는 스프링 프레임워크의 계층별로 나위는데, 다음과 같은 애노테이션을 이용할 수 있다.
- @Component: 클래스를 빈으로 정의하는 스테레오 타입 애노테이션으로 org.springframework.stereotype.Component를 임포트해야 한다.
- @Repository: 모든 저장소 클래스에 @Repository 애노테이션을 지정해 저장소 클래스임을 나타낸다.
  저장소 클래스는 애플리케이션의 영속성 계층에서 모든 데이터 접근 로직을 포함하는 데이터 접근 객체 DAO의 역할을 한다.
  @Repository 애노테이션을 사용하려면 org.springframework.stereotype.Repsitory를 임포트해야 한다.
- @Service: 모든 비즈니스 로직을 포함하는 서비스 클래스에 @Service 애노테이션을 지정하며, 이 애노테이션을 사용하려면
  org.springframework.stereotype.Service를 임포트해야 한다.
- @Controller: @Controller 애노테이션이 지정된 클래스는 스프링의 컨트롤러 타입임을 나타낸다.
  즉, 해당 클래스가 스프링 MVC의 컨트롤러 역할을 함을 나타내며, 이 애노테이션을 사용하려면 org.springframework.stereotype.Controller를 임포트해야 한다.

### Q2. 이벤트 처리의 구성요소를 설명하라.
이러한 이벤트 처리는 ApplicationContext가 제공하는 중요한 기능으로, 크게 세 가지 핵심 요소로 구성된다.
- ApplicationListener: 이벤트를 리스닝하는 모든 클래스는 ApplicationListener 인터페이스를 구현해야 한다.
  ApplicationEvent가 발생할 때마다 ApplicationListener를 구현한 빈은 통지를 받게 된다.
  이 인터페이스를 구현할 때는 onApplicationEvent() 메소드를 구현해야 한다.
- ApplicationEventPublisher: 이벤트를 발생시키는 모든 클래스는 ApplicationEventPublisher 인터페이스를 구현해야 한다.
  애플리케이션 이벤트 발생 클래스의 publishEvent() 메소드를 호출하면 빈에서 이벤트를 발생시킬 수 있다.
- ApplicationEvent: 사용자 정의 이벤트를 만들고 기능을 추가하고 이벤트에 대한 추가적인 메타 데이터를 설정할 때 이 클래스를 사용한다.

### Q3. 스프링 표현 언어는 무엇인가?
스프링 표현 언어 SpEL, Spring Expression Language는 런타임시에 객체 그래프를 질의하고 조작할 수 있는 기능을 지원하는 강력한 표현 언어다.
SpEL을 이용하면 프로퍼티의 값을 동적으로 평가하고 IoC 컨트롤러에 설정된 값으로 사용할 수 있다.
SpEL은 수학적 연산자와 논리적 연산자, 관계 연산자를 지원하며, matches 연산자를 이용하면 정규표현식도 사용할 수 있다.
SpEL은 런타임 시에 동적인 빈 와이어링을 지원하며, 의존성 주입에 사용할 적한한 빈이나 값을 골라낸다.
또한 빈이나 빈 프로퍼티를 다른 빈에 주입하거나, 다른 빈의 빈 메소드를 호출할 때도 SpEL을 이용할 수 있다.
SpEL의 기능은 다음과 같다.
- 빈의 ID를 이용해 빈을 참조한다.
- 빈에 메소드와 프로퍼티를 주입한다.
- 값에 수학적 연산자와 논리적 연산자, 관계 연산자를 이용한다.
- 정규표현식을 매칭한다.
- 컬렉션을 조작한다.

## 10장. 스프링의 관점지향 프로그래밍
### Q1. 관점지향 프로그래밍이란 무엇인가?
AOP란 OOP에서는 수행하기 어려운 횡단 관심 분리르 위한 전도 유망한 기술이다.
AOP는 지원 기능으로부터 메인 프로그램 비즈니스 로직을 분리하는 프로그래밍 패러다임을 의미한다.
스프링 프레임워크에서 AOP는 선언형 엔터프라이즈 서비스를 제공한다.
여기서 AOP의 애플리케이션 객체는 비즈니스 로직을 수행할 뿐 로깅, 보안, 감사, 잠금, 이벤트 처리와 같은 타 시스템의 애스펙트에 대한 책임을 지지 않는다.
AOP는 보안 서비스, 트랜잭션 관리 서비스 등과 같은 미들웨어 서비스를 스프링 애플리케이션에 적용하는 방법론이다.

### Q2. 스프링에서 관심, 어드바잇, 애스펙트, 조인포인트, 포인트컷이란 무엇인가?
관심은 기능에 따라 구분되는 시스템의 부분을 의미한다.
관심은 핵심 또는 횡단이 될 수 있으며, 핵심 애스펙트는 주요 요구사항에 대한 특정 기능을 나타내고,
횡단 관심은 시스템 전체 애스펙트로, 로깅, 보안 등과 같은 부차적인 요구사항을 위한 기능을 나타낸다.
- 어드바이스 Advice: 어드바이스란 조인포인트에서 수행되는 코드를 의미한다. 어드바이스는 시스템 전체 애스펙트에 API 호출을 제공한다.
- 애스펙트 Aspect: 애스펙트는 포인트컷과 어드바이스의 결합된 애스펙트로써, 애플리케이션에 포함되는 횡단 관심 기능이다.
- 조인포인트 Joinpoint: 조인포인트란 클래스 초기화, 객체 인스턴스화, 메소드 호출, 필드 접근, 예외 발생과 같은 애플리케이션 실행 흐름에서의 특정 포인트를 의미한다.
  애플리케이션에 새로운 동작을 추가하기 위해 조인포인트에 애스펙트 코드를 추가할 수 있다. 횡단 관심은 조인포인트 전/후에 AOP에 의해 자동으로 추가된다.
- 포인트 컷 Pointcut: 포인트컷은 어드바이스가 적용될 곳을 명시하는 조인포인트의 콜렉션을 의미한다.

### Q3. 어드바이스 유형은 무엇인가?
어드바이스의 종류는 다음과 같다.
- Before advice: 조인포인트 전에 실행된다. 메소드가 호출되기 전에 어드바이저 코드를 실행할 수 있다.
- After-returning advice: 메소드가 성공적으로 실행을 마친 후에 적용된다.
- Throws advice: 실행 중에 메소드가 예외를 발생시킬 때 적용된다.
- After(finally) advice: 이름에서 알 수 있듯이 조인 포인트 메소드 수행이 정상적으로 종료된 후에 또는 예외가 발생된 후에 실행된다.
- Around advice: 메소드 실행 전후에 적용된다.

### Q4. 스프링에서의 위빙 weaving이란 무엇인가.
위빙이란 적절한 지점에 애플리케이션으로 애스펙트를 삽입하는 처리 과정이다.
위빙은 클래스의 생명주기 내 여러 단계에서 발생한다.
- 컴파일 타임: 컴파일 중 어드바이스의 바이트 코드를 조인포인트로 삽입하는 것을 컴파일 타임 위빙이라 부른다.
- 클래스로드 타임: 클래스가 로드되는 시점에 바이트 코드를 삽입한다. 조인포인트에 어드바이스 코드를 삽입하기 위해 클래스가 로드되는 동안 바이트 코드가 로드된 클래스로 삽입된다.
- 런타임-스프링 방식: 타깃 객체는 프록시 빈으로 보호되며, 프록시 빈은 스프링 프레임워크에 의해 생성된다.
  호출자가 타깃 빈의 메소드를 호출하면 스프링 프레임워크는 프록시를 호출해 타깃 메소드에 어드바이스를 적용한다.
  메소드 실행이 끝나면, 요청이 있을 경우 스프링은 타깃 메소드에 다시 어드바이스를 적용하고 응답을 호출자에게 반환한다.

스프링은 위와 같은 종류의 위빙을 사용한다. 런타임 위빙이 코드를 깔끔하게 유지하는데 효과적인다.
